Decisión: Reemplazar browser getUserMedia por MJPEG stream del detector.

Contexto: El navegador y el detector competían por /dev/video0. getUserMedia era poco confiable (timeouts, sin errores).

Qué se hizo:
1. detector.py: default SOURCE → webcam, agregado server MJPEG en puerto 8080 (/stream)
2. docker-compose.yml: detector port 8081:8080
3. detector/Dockerfile: ENV PYTHONUNBUFFERED=1, EXPOSE 8080
4. dashboard/nginx.conf: proxy /stream → detector:8080/stream
5. dashboard/public/index.html: reemplazado <video> por <img src='/stream'>
6. dashboard/public/app.js: eliminado todo el código de getUserMedia
7. .env / .env.example: SOURCE=webcam
8. .env.example: restaurado con SOURCE=webcam

Comprobación:
- Detector abrió webcam (30 FPS), MJPEG stream funcional
- Detecta movimiento real (pixels=16466, high severity)
- Eventos enviados al backend (HTTP 201) y almacenados en PostgreSQL
- Stream MJPEG accesible vía puerto 8081 y nginx proxy

Próximos pasos: El usuario debe asegurarse de que /dev/video0 esté disponible y ejecutar 'docker compose up --build -d'.

## agent

Fix crítico: variable global shadoweada en detector.py

Bug: En main(), la línea '_latest_jpeg = jpeg_buf.tobytes()' creaba una variable LOCAL porque Python requiere 'global _latest_jpeg' para asignar a una variable del módulo. El MJPEG Handler leía la variable global que nunca se actualizaba (b'').

Síntoma: El stream MJPEG devolvía HTTP 200 con Content-Type correcto pero 0 bytes de datos. El dashboard mostraba fondo negro.

Fix: Agregado 'global _latest_jpeg' al inicio de main().

Verificación: Stream ahora sirve frames JPEG (~500KB/s), con boundaries '--frame', JFIF headers válidos.

## agent

Fix crítico: Redis timeout impedía que eventos llegaran al WebSocket.

Causa raíz: get_redis() creaba una NUEVA conexión cada vez, con
timeouts por defecto (~10s). El worker de Redis usaba pubsub.listen()
que timeout al estar idle, ciclando: connect → subscribe → timeout →
reconnect → ... Los eventos se guardaban en PostgreSQL pero no llegaban
al dashboard.

Fixes:
1. redis_client.py: conexión singleton compartida entre worker y API.
   Config: socket_keepalive=True, socket_timeout=30s,
   health_check_interval=15s, retry_on_timeout=True.
2. main.py: pre-warm Redis en startup, backoff exponencial (1→2→4→...→30s).
3. events.py: reusa la conexión compartida, log warning si falla publish.
4. app.js: defensivo con isNaN(ts.getTime()), si timestamp inválido
   muestra hora actual en vez de 'Invalid Date'.

Verificación: 0 TimeoutErrors tras el fix, 3 eventos creados con 201,
WebSocket conectado.
Commit: db24692
